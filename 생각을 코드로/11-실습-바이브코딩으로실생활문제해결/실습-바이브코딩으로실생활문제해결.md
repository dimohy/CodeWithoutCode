# 11주차: 실습 - 바이브 코딩으로 실생활 문제 해결

## 개요

### 지난 주 복습: 프롬프트 작성의 예술

지난 10주차에서 우리는 중요한 기술을 마스터했습니다. 바로 **효과적인 프롬프트 작성법**입니다. 좋은 프롬프트의 3요소(문제 정의, 예상 결과, 제약 조건)를 배웠고, "계산기 만들어줘"같은 모호한 요청을 "두 숫자를 입력받아 사칙연산하고 0으로 나누면 오류 보여주는 간단한 계산기"처럼 구체적으로 바꾸는 법을 연습했습니다.

특히 **Agent 모드를 활용한 복잡한 작업 위임**도 배웠습니다. Chat 창 하단에서 Agent 모드를 선택하면 AI가 자율적으로 여러 파일을 생성하고 수정하며, 프로젝트 전체를 이해하고 작업합니다. `@workspace` 멘션을 함께 사용하면 더욱 강력한 컨텍스트를 제공할 수 있습니다. 또한 Copilot의 응답을 평가하고 개선을 요청하는 법, 반복적 대화를 통해 점차 완성도를 높이는 방법도 익혔습니다.

이제 여러분은 **사고하는 법**(4대 원리), **표현하는 법**(프롬프트 작성), **협업하는 법**(Agent 활용)을 모두 갖췄습니다. 이 모든 것을 통합할 시간입니다.

### 이번 주: 여러분의 문제, 여러분의 해결책

지금까지 여러분은 컴퓨팅 사고의 4대 원리를 배우고, GitHub Copilot과 대화하는 방법을 익혔으며, 다양한 실습을 통해 바이브 코딩의 기초를 다졌습니다. 이제 그 모든 것을 하나로 모아 실제 생활에서 마주치는 문제들을 해결해볼 시간입니다.

이번 주차는 여러분이 일상에서 겪는 진짜 문제들을 바이브 코딩으로 풀어보는 실습 중심 챕터입니다. 단순히 예제를 따라하는 것이 아니라, 여러분 자신의 문제를 선택하고, 4대 원리로 분석하고, GitHub Copilot과 협업하여 해결책을 만들어가는 전 과정을 경험하게 됩니다. 

이 과정에서 여러분은 바이브 코딩이 단순히 프로그램을 만드는 기술이 아니라, 생각하는 방식 자체를 바꾸는 도구라는 것을 깨닫게 될 것입니다. 코드를 직접 작성하지 않더라도, 명확하게 생각하고 정확하게 요청하면 원하는 결과를 얻을 수 있다는 것을 체감하게 됩니다.

**이번 주차의 학습 목표:**
- 실생활 문제를 바이브 코딩으로 해결할 수 있는 문제로 변환하기
- 4대 원리(분해, 패턴 인식, 추상화, 알고리즘적 사고)를 통합 적용하기
- GitHub Copilot과 효과적으로 협업하여 실용적인 솔루션 만들기
- 다양한 유형의 문제에 대한 접근 패턴 익히기

## 실생활 문제를 바이브 코딩 문제로 보기

우리는 매일 크고 작은 문제들을 만납니다. 아침에 일어나서 준비할 것들이 너무 많아 늦는 문제, 이메일이 너무 많아서 중요한 것을 놓치는 문제, 가계부를 써야 하는데 귀찮아서 미루는 문제까지. 이런 문제들은 언뜻 보기에는 '프로그래밍'과는 아무 관련이 없어 보입니다.

하지만 바이브 코딩의 관점에서 보면, 이 모든 문제는 해결 가능한 문제입니다. 핵심은 "이 문제를 어떻게 컴퓨터가 도와줄 수 있을까?"가 아니라 "이 문제를 어떻게 명확하게 정리하고, 단계별로 풀어갈까?"입니다.

### 문제를 바라보는 새로운 시각

전통적으로 문제를 해결한다는 것은 직접 손으로, 몸으로 해결하는 것을 의미했습니다. 가계부 문제가 있다면 노트를 사서 직접 쓰고, 이메일이 많다면 하나하나 읽으며 분류하고, 일정이 복잡하다면 달력에 빼곡히 적어넣었죠.

바이브 코딩은 이런 문제들에 대해 다르게 접근합니다. "내가 직접 할 필요가 있을까? 컴퓨터에게 어떻게 설명하면 도움을 받을 수 있을까?"라고 묻습니다. 여기서 중요한 것은 '설명'입니다. 여러분이 문제를 명확하게 설명할 수 있다면, GitHub Copilot은 그것을 실행 가능한 형태로 만들어줄 수 있습니다.

### 해결 가능한 문제의 특징

모든 문제가 바이브 코딩으로 해결되는 것은 아닙니다. 하지만 놀랍게도 생각보다 많은 문제들이 해결 가능합니다. 바이브 코딩으로 접근하기 좋은 문제들은 다음과 같은 특징을 가지고 있습니다:

**반복적인 작업이 있는 문제:** 매일 똑같은 형식의 보고서를 작성하거나, 비슷한 패턴의 데이터를 정리하거나, 규칙적으로 확인해야 하는 정보가 있다면 바이브 코딩이 큰 도움이 됩니다. 한 번 GitHub Copilot에게 요청하는 방법을 배우면, 매번 같은 일을 반복하지 않아도 됩니다.

**규칙이 명확한 문제:** "이런 조건이면 이렇게 하고, 저런 조건이면 저렇게 한다"처럼 규칙이 분명한 문제는 바이브 코딩의 완벽한 대상입니다. 예를 들어 "50,000원 이상 지출은 빨간색으로 표시"같은 규칙은 GitHub Copilot에게 정확하게 전달할 수 있습니다.

**데이터를 다루는 문제:** 많은 양의 정보를 정리하거나, 분류하거나, 요약하거나, 변환해야 하는 문제라면 바이브 코딩이 특히 강력합니다. 사람이 하면 몇 시간 걸릴 일을 몇 분 안에 끝낼 수 있습니다.

**형식을 변환하는 문제:** 엑셀 데이터를 보기 좋은 표로 만들거나, 여러 파일의 정보를 하나로 합치거나, 특정 형식으로 문서를 만들어야 할 때도 바이브 코딩이 유용합니다.

반대로, 창의적 판단이 중요하거나, 감정적 교감이 필요하거나, 물리적 행동이 필요한 문제는 바이브 코딩만으로는 완전히 해결하기 어렵습니다. 하지만 이런 문제들도 일부는 바이브 코딩으로 보조할 수 있습니다. 예를 들어 창의적인 글쓰기 자체는 사람이 하더라도, 아이디어 정리나 초안 작성은 GitHub Copilot이 도와줄 수 있습니다.

### 문제를 바이브 코딩 언어로 번역하기

실생활 문제를 바이브 코딩으로 해결하려면 한 단계의 '번역' 과정이 필요합니다. 이것은 컴퓨팅 사고의 4대 원리를 적용하는 과정이기도 합니다.

예를 들어 "이메일이 너무 많아서 정리가 안 된다"는 막연한 문제가 있다고 해봅시다. 이것을 바이브 코딩 언어로 번역하면 이렇게 됩니다:

1. **분해:** "정리가 안 된다"를 구체적으로 나누어봅니다. 읽지 않은 이메일과 읽은 이메일을 구분하기, 중요한 이메일과 덜 중요한 이메일 분류하기, 업무용과 개인용 나누기 등으로 나눌 수 있습니다.

2. **패턴 인식:** 어떤 이메일이 중요한지 패턴을 찾습니다. 특정 발신자의 이메일, 특정 키워드가 포함된 이메일, 특정 시간대의 이메일 등에서 패턴이 보일 수 있습니다.

3. **추상화:** 핵심만 남깁니다. 모든 이메일을 완벽하게 분류할 필요는 없을 수도 있습니다. "일단 급한 것만 먼저 골라내자"처럼 핵심 목표를 명확히 합니다.

4. **알고리즘적 사고:** 단계를 정리합니다. "먼저 읽지 않은 이메일 목록을 가져오고, 그 중에서 특정 키워드가 있는 것을 찾고, 그것들을 별도 폴더로 옮긴다" 같은 순서를 생각합니다.

이렇게 번역하고 나면, GitHub Copilot에게 무엇을 요청해야 할지 훨씬 명확해집니다. "이메일 정리 좀 해줘"보다는 "읽지 않은 이메일 중에서 '긴급' 또는 '회의'라는 단어가 포함된 이메일을 찾아서 '중요' 폴더로 옮기는 작업을 도와줘"라고 요청할 수 있게 됩니다.

## 문제 유형별 접근 패턴

실생활 문제들은 다양하지만, 바이브 코딩으로 접근할 때는 몇 가지 전형적인 패턴으로 분류할 수 있습니다. 각 패턴마다 효과적인 접근 방법이 있고, GitHub Copilot과 협업하는 방식도 조금씩 다릅니다. 지금부터 대표적인 문제 유형들을 살펴보고, 각각에 어떻게 접근하면 좋을지 알아보겠습니다.

### 데이터 정리 및 분석 문제

이것은 아마도 가장 흔하게 마주치는 유형일 것입니다. 엑셀 파일에 흩어진 데이터를 정리하거나, 여러 출처의 정보를 하나로 모으거나, 특정 조건에 맞는 항목만 골라내야 하는 상황들이죠.

**이런 문제의 특징:**
- 처리해야 할 데이터가 많아서 손으로 하기엔 시간이 오래 걸림
- 규칙이 명확해서 설명하기 쉬움
- 한 번 해결하면 비슷한 상황에서 반복 사용 가능

**바이브 코딩 접근법:**

먼저 데이터가 어떤 형태인지 파악합니다. 엑셀 파일인가요, 텍스트 파일인가요, 아니면 여러 문서에 흩어져 있나요? 그리고 최종적으로 어떤 형태로 만들고 싶은지 생각합니다.

다음으로 "어떤 것을 어떻게"를 명확히 합니다. 예를 들어:
- "이 중에서 어떤 것을 골라낼까?" (조건)
- "어떤 순서로 정렬할까?" (정렬)
- "어떤 기준으로 묶을까?" (분류)
- "어떤 계산을 할까?" (집계)

마지막으로 GitHub Copilot에게 단계적으로 요청합니다. 한 번에 모든 것을 요청하기보다는, 작은 단계로 나누어 하나씩 확인하면서 진행하는 것이 좋습니다.

**GitHub Copilot과의 대화 예시:**

여러분이 가계부 데이터가 담긴 엑셀 파일을 정리하고 싶다고 해봅시다. 이렇게 대화를 시작할 수 있습니다:

"엑셀 파일에 날짜, 항목, 금액이 있어. 이 중에서 금액이 50,000원 이상인 항목만 골라서 새로운 시트로 만들고 싶어."

GitHub Copilot이 방법을 제시하면, 결과를 확인하고 다음 단계로 진행합니다:

"좋아. 이제 이 항목들을 날짜순으로 정렬해줘."

"완벽해. 마지막으로 각 달별로 합계를 계산해서 표로 만들어줘."

이렇게 단계적으로 진행하면, 복잡한 데이터 정리 작업도 자연스럽게 대화하듯 해결할 수 있습니다.

### 반복 작업 자동화 문제

매일, 매주, 매월 똑같은 작업을 반복하는 것만큼 지루한 일이 또 있을까요? 보고서 양식에 데이터를 채워 넣거나, 같은 형식의 파일을 계속 만들거나, 정기적으로 특정 정보를 확인하는 일들이 여기에 해당합니다.

**이런 문제의 특징:**
- 매번 비슷한 패턴이 반복됨
- 실수하기 쉽지만 정확성이 중요함
- 시간은 많이 걸리지만 생각은 별로 필요 없음

**바이브 코딩 접근법:**

반복 작업을 자동화할 때는 먼저 "정확히 무엇이 반복되는가"를 파악해야 합니다. 같은 것처럼 보여도 매번 조금씩 다를 수 있습니다. 완전히 똑같은 부분과 매번 바뀌는 부분을 구분하는 것이 중요합니다.

똑같은 부분은 '템플릿'으로 만들 수 있습니다. 매번 바뀌는 부분은 '변수'로 생각할 수 있습니다. 이것이 바로 추상화입니다. "이 부분은 항상 같고, 저 부분만 매번 다르다"라고 정리하는 거죠.

그 다음 GitHub Copilot에게 "이런 템플릿에 이런 값들을 넣어서 만들어줘"라고 요청하면 됩니다.

**GitHub Copilot과의 대화 예시:**

매주 회의록을 작성해야 하는데, 형식은 항상 똑같다고 해봅시다:

"매주 회의록을 작성하는데, 항상 이런 형식이야:
- 회의 날짜: [날짜]
- 참석자: [이름들]
- 주요 안건: [안건 목록]
- 결정 사항: [결정 내용]
- 다음 회의: [날짜]

이 형식으로 문서를 자동으로 만들고 싶어. 내가 정보만 입력하면 완성되게."

GitHub Copilot은 이 템플릿을 만들고, 여러분이 필요한 정보만 입력하면 자동으로 완성된 회의록을 만들어주는 방법을 제시할 것입니다.

### 일정 및 작업 관리 문제

해야 할 일이 너무 많거나, 일정이 복잡하거나, 우선순위를 정하기 어려울 때가 있습니다. 이런 문제는 정보를 체계적으로 정리하고 시각화하면 훨씬 관리하기 쉬워집니다.

**이런 문제의 특징:**
- 여러 일정과 작업이 얽혀 있음
- 우선순위나 마감일이 중요함
- 상황이 계속 변하므로 유연성이 필요함

**바이브 코딩 접근법:**

일정 관리 문제는 먼저 "보고 싶은 것"을 명확히 하는 것이 중요합니다. 모든 일정을 다 보고 싶은가요, 아니면 이번 주 할 일만 보고 싶은가요? 마감일이 가까운 것부터 보고 싶은가요, 아니면 중요도 순으로 보고 싶은가요?

그리고 정보를 어떻게 입력할지 생각합니다. 가장 간단한 방법은 텍스트 파일이나 간단한 표 형식일 수 있습니다. 복잡할 필요는 없습니다. 여러분이 편하게 입력할 수 있는 형식이 가장 좋은 형식입니다.

GitHub Copilot에게는 "이런 정보를 이런 식으로 보여줘"라고 요청하면 됩니다.

**GitHub Copilot과의 대화 예시:**

"할 일 목록이 텍스트 파일로 정리되어 있어. 각 항목마다 [중요], [보통], [낮음] 같은 우선순위 표시가 있고, 마감일도 적혀 있어. 이걸 마감일 순으로 정렬하되, 같은 날짜면 중요도 순으로 보여주는 표를 만들고 싶어."

"좋아. 이제 오늘 날짜 기준으로 3일 이내 마감인 것만 따로 뽑아서 '급한 일' 목록을 만들어줘."

이렇게 하면 복잡한 할 일 목록도 한눈에 파악할 수 있는 형태로 정리됩니다.

### 콘텐츠 생성 및 변환 문제

글을 쓰거나, 자료를 만들거나, 형식을 바꾸는 작업들도 바이브 코딩으로 접근할 수 있습니다. 물론 창의적인 내용 자체는 여러분이 결정해야 하지만, 형식을 갖추거나 초안을 만드는 데는 GitHub Copilot이 큰 도움을 줄 수 있습니다.

**이런 문제의 특징:**
- 정해진 형식이나 구조가 있음
- 비슷한 내용을 반복적으로 만들어야 함
- 형식 변환이 번거로움

**바이브 코딩 접근법:**

콘텐츠 생성에서 가장 중요한 것은 "무엇을 담을 것인가"와 "어떤 형태로 표현할 것인가"를 분리하는 것입니다. 내용은 여러분이 생각하고, 형태는 GitHub Copilot이 만들어주는 식으로 협업할 수 있습니다.

예를 들어 발표 자료를 만든다면, 핵심 메시지와 포함할 정보는 여러분이 정하고, 슬라이드 구조나 형식은 GitHub Copilot에게 맡길 수 있습니다.

**GitHub Copilot과의 대화 예시:**

"다음 주 팀 회의에서 프로젝트 진행 상황을 발표해야 해. 슬라이드 5장 정도로 만들고 싶은데, 각 슬라이드에 들어갈 구조를 제안해줘. 내용은 프로젝트 목표, 현재 진행률, 주요 성과, 직면한 문제, 다음 단계야."

GitHub Copilot이 구조를 제안하면, 여러분은 각 슬라이드에 실제 내용을 채워 넣으면 됩니다.

"좋아. 이제 각 슬라이드의 제목과 핵심 포인트를 마크다운 형식으로 만들어줘."

이런 식으로 골격은 빠르게 만들고, 살은 여러분이 붙이는 협업이 가능합니다.

### 정보 수집 및 요약 문제

인터넷이나 여러 문서에서 정보를 모으고, 중요한 내용만 추려내고, 보기 좋게 정리하는 작업도 바이브 코딩의 좋은 대상입니다.

**이런 문제의 특징:**
- 정보가 여러 곳에 흩어져 있음
- 중요한 것과 덜 중요한 것을 구분해야 함
- 같은 정보를 다른 형식으로 재구성해야 함

**바이브 코딩 접근법:**

먼저 "어떤 정보를 어디서 가져올 것인가"를 정합니다. 웹페이지인가요, 문서 파일인가요, 이메일인가요? 그리고 "어떤 형식으로 정리할 것인가"를 생각합니다.

GitHub Copilot에게는 정보의 출처와 원하는 결과물의 형태를 명확히 설명하면 됩니다. "이런 정보들을 모아서 이런 형식으로 만들어줘"처럼요.

**GitHub Copilot과의 대화 예시:**

"여러 개의 텍스트 파일에 회의 내용이 흩어져 있어. 각 파일에서 '결정사항:' 다음에 나오는 내용만 뽑아서 하나의 목록으로 만들고 싶어."

"잘 됐어. 이제 이 결정사항들을 날짜순으로 정렬하고, 각 항목 앞에 어느 회의에서 나온 건지 표시해줘."

이렇게 하면 흩어진 정보를 체계적으로 모을 수 있습니다.

## 실습: 나만의 문제 해결하기

이제 이론을 넘어 실제로 여러분의 문제를 해결해볼 시간입니다. 이 실습에서는 여러분이 직접 문제를 선택하고, 4대 원리를 적용하여 분석하고, GitHub Copilot과 협업하여 해결하는 전 과정을 경험하게 됩니다.

### 단계 1: 문제 선택하기

먼저 여러분이 실제로 해결하고 싶은 문제를 하나 선택하세요. 너무 거창할 필요는 없습니다. 일상에서 "이게 좀 불편한데", "이거 자동으로 안 될까?" 싶었던 것이면 충분합니다.

**좋은 첫 문제의 예시:**
- 여러 폴더에 흩어진 사진 파일을 날짜별로 정리하고 싶다
- 매달 받는 전화요금 청구서를 정리해서 지출 패턴을 보고 싶다
- 메모해둔 아이디어들을 주제별로 분류하고 싶다
- 독서 목록을 장르별, 읽은/안 읽은 상태별로 정리하고 싶다
- 운동 기록을 주간별로 요약해서 보고 싶다

문제를 선택할 때 이렇게 자문해보세요:
- 이 문제를 해결하면 실제로 내 생활이 편해질까?
- 명확한 규칙이나 패턴이 있을까?
- 데이터나 정보를 다루는 문제일까?

이 질문에 "예"라고 답할 수 있다면, 바이브 코딩으로 접근하기 좋은 문제입니다.

### 단계 2: 4대 원리로 문제 분석하기

문제를 선택했다면, 이제 컴퓨팅 사고의 4대 원리를 적용하여 분석해봅시다. 종이나 메모장을 준비하고, 각 원리별로 생각을 정리해보세요.

**분해(Decomposition):**

큰 문제를 작은 문제로 나눕니다. "정리하고 싶다"는 막연한 목표를 구체적인 단계들로 쪼개는 것입니다.

예를 들어 "사진 파일을 날짜별로 정리"한다면:
- 모든 사진 파일을 찾기
- 각 사진의 촬영 날짜 확인하기
- 날짜별 폴더 만들기
- 사진을 해당 폴더로 옮기기

이렇게 나누면 각 단계가 훨씬 명확해집니다. GitHub Copilot에게도 한 번에 모든 것을 요청하는 것보다, 단계별로 요청하는 것이 더 정확한 결과를 얻을 수 있습니다.

**패턴 인식(Pattern Recognition):**

반복되는 규칙이나 패턴을 찾습니다. 모든 데이터가 똑같은 방식으로 처리될 수 있을까요? 아니면 몇 가지 유형이 있을까요?

사진 정리 예시에서:
- 모든 사진 파일이 같은 형식일까? (JPG, PNG 등)
- 촬영 날짜 정보가 파일 이름에 있을까, 아니면 파일 속성에 있을까?
- 날짜 형식이 어떻게 되어 있을까? (2025-11-13, 20251113 등)

패턴을 파악하면 GitHub Copilot에게 더 정확한 정보를 줄 수 있습니다.

**추상화(Abstraction):**

핵심만 남기고 불필요한 세부사항은 제거합니다. 모든 것을 완벽하게 할 필요는 없습니다. 정말 중요한 것이 무엇인지 결정하세요.

사진 정리에서:
- 년-월-일까지 다 나눌까, 아니면 월 단위로만 나눌까?
- 정확히 같은 날짜끼리만 묶을까, 아니면 주 단위로 묶을까?
- 파일 이름은 그대로 둘까, 아니면 새로 지을까?

처음에는 간단하게 시작하는 것이 좋습니다. 나중에 필요하면 더 세밀하게 만들 수 있습니다.

**알고리즘적 사고(Algorithmic Thinking):**

순서를 정합니다. 어떤 단계를 먼저 해야 할까요? 어떤 것은 이전 단계가 끝나야 할 수 있을까요?

사진 정리의 순서:
1. 먼저 모든 사진 파일의 위치를 파악
2. 그 다음 각 파일의 날짜 정보 읽기
3. 필요한 폴더들 만들기
4. 마지막으로 파일 이동하기

이 순서가 중요한 이유는, 폴더를 먼저 만들지 않으면 파일을 옮길 곳이 없기 때문입니다.

### 단계 3: GitHub Copilot에게 요청하기

4대 원리로 분석했다면, 이제 GitHub Copilot과 대화를 시작할 준비가 된 것입니다. 분석한 내용을 바탕으로 차근차근 요청해봅시다.

**첫 번째 요청: 상황 설명**

처음에는 전체 상황을 설명합니다. 무엇을 하고 싶은지, 현재 어떤 상태인지 GitHub Copilot이 이해할 수 있게 알려주세요.

"여러 폴더에 사진 파일들이 흩어져 있어. JPG 파일인데, 각 파일은 촬영 날짜 정보를 가지고 있어. 이 사진들을 년-월 폴더로 정리하고 싶어. 예를 들면 2025년 11월에 찍은 사진은 '2025-11' 폴더에 모으고 싶어."

**두 번째 요청: 첫 단계 실행**

전체를 한 번에 하려고 하지 말고, 첫 단계부터 시작합니다.

"먼저 특정 폴더에서 모든 JPG 파일 목록을 가져오는 것부터 도와줘."

GitHub Copilot이 방법을 제시하면, 그것을 실행해보고 결과를 확인합니다. 원하는 대로 되었나요? 파일 목록이 제대로 나왔나요?

**세 번째 요청: 단계적 진행**

첫 단계가 성공했다면 다음 단계로 진행합니다.

"좋아. 이제 각 파일의 촬영 날짜를 확인하는 방법을 알려줘."

"완벽해. 이제 날짜 정보를 바탕으로 '2025-11' 같은 형식의 폴더를 만들고, 해당 파일을 옮기는 것을 도와줘."

**네 번째 요청: 확인 및 개선**

기본적인 기능이 작동하면, 필요한 부분을 개선합니다.

"잘 되는데, 폴더가 이미 있으면 오류가 나네. 폴더가 없을 때만 만들도록 수정해줘."

"혹시 날짜 정보가 없는 사진은 '날짜미상' 폴더로 따로 모을 수 있을까?"

### 단계 4: 결과 확인 및 테스트

GitHub Copilot이 제시한 방법으로 실제로 작업을 수행해봅니다. 처음에는 테스트용 파일 몇 개로만 시도해보는 것이 좋습니다. 모든 파일에 한 번에 적용했다가 문제가 생기면 되돌리기 어려울 수 있으니까요.

**테스트 체크리스트:**
- 원하는 결과가 나왔나요?
- 예상치 못한 문제는 없나요?
- 모든 경우를 처리하나요? (예: 날짜 정보가 없는 파일, 이미 같은 이름의 파일이 있는 경우 등)
- 결과를 되돌릴 수 있나요? (원본 파일이 삭제되지 않았나요?)

문제가 있다면 GitHub Copilot에게 다시 설명하고 수정을 요청합니다. "이런 경우에 이런 문제가 생기는데, 어떻게 하면 좋을까?"

### 단계 5: 재사용 가능하게 만들기

한 번 해결한 문제는 나중에 또 활용할 수 있습니다. GitHub Copilot에게 이렇게 요청해볼 수 있습니다:

"이 방법을 다음에도 쉽게 사용할 수 있게, 폴더 경로만 바꾸면 실행할 수 있도록 만들어줘."

또는:

"이 과정을 단계별로 정리해서, 다음에 다른 파일 정리할 때도 참고할 수 있게 해줘."

이렇게 하면 비슷한 문제를 만났을 때 처음부터 다시 시작하지 않아도 됩니다.

## 다양한 실습 예제

여러분이 직접 문제를 해결하는 과정을 도와드리기 위해, 몇 가지 다른 유형의 예제를 더 살펴보겠습니다. 각 예제는 서로 다른 접근 방식을 보여줍니다.

### 예제 1: 가계부 데이터 분석하기

**문제 상황:**
매달 가계부를 쓰고 있는데, 엑셀 파일에 날짜, 카테고리, 내용, 금액이 기록되어 있습니다. 몇 달치 데이터가 쌓였는데, 어디에 돈을 가장 많이 쓰는지 한눈에 보고 싶습니다.

**4대 원리 적용:**

분해:
- 데이터를 카테고리별로 나누기
- 각 카테고리의 합계 계산하기
- 월별로 비교하기
- 시각적으로 표현하기

패턴 인식:
- 매달 반복되는 고정 지출 찾기
- 특정 카테고리의 지출 증가/감소 패턴 파악하기
- 요일별 지출 패턴 있는지 확인하기

추상화:
- 세부 항목보다는 카테고리별 총액에 집중
- 모든 거래가 아니라 큰 금액 위주로 보기
- 월별 트렌드만 파악하기

알고리즘적 사고:
1. 엑셀 파일에서 데이터 읽기
2. 카테고리별로 그룹화하기
3. 각 카테고리의 합계 계산하기
4. 결과를 표나 그래프로 만들기

**GitHub Copilot과의 협업:**

첫 대화는 이렇게 시작할 수 있습니다:

"엑셀 파일에 가계부 데이터가 있어. A열은 날짜, B열은 카테고리(식비, 교통비, 통신비 등), C열은 내용, D열은 금액이야. 이 데이터를 카테고리별로 묶어서 각각 얼마나 지출했는지 합계를 보고 싶어."

GitHub Copilot이 방법을 제시하면 실행해보고, 결과를 바탕으로 다음 요청을 합니다:

"좋아. 이제 이걸 큰 금액 순서로 정렬해줘. 그리고 전체 지출 중 각 카테고리가 차지하는 비율도 계산해줘."

"완벽해. 이제 이 정보를 막대 그래프로 만들 수 있을까?"

이런 식으로 단계적으로 진행하면, 복잡한 데이터 분석도 대화하듯 자연스럽게 완성할 수 있습니다.

**핵심 포인트:**
- 데이터의 구조를 정확히 설명하기 (어느 열에 무엇이 있는지)
- 한 번에 하나씩 요청하기 (합계 → 정렬 → 비율 → 그래프)
- 중간 결과를 확인하며 진행하기

### 예제 2: 회의록 템플릿 자동화

**문제 상황:**
매주 팀 회의를 하고 회의록을 작성하는데, 형식은 항상 똑같습니다. 날짜, 참석자, 안건, 결정사항, 다음 회의 일정 등이 정해진 순서로 들어갑니다. 매번 이 형식을 처음부터 만드는 게 귀찮습니다.

**4대 원리 적용:**

분해:
- 고정된 부분(제목, 섹션 이름) 파악하기
- 가변적인 부분(날짜, 참석자 등) 파악하기
- 입력받을 정보 목록 만들기
- 완성된 문서 형식 정의하기

패턴 인식:
- 매주 반복되는 구조 찾기
- 참석자 명단의 형식 패턴
- 안건 항목의 공통 형식

추상화:
- 핵심 정보만 템플릿에 포함
- 부가적인 장식은 제거
- 필수 항목과 선택 항목 구분

알고리즘적 사고:
1. 필요한 정보를 입력받기
2. 템플릿에 정보 채워 넣기
3. 완성된 문서 생성하기
4. 파일로 저장하기

**GitHub Copilot과의 협업:**

"매주 회의록을 쓰는데, 항상 이런 형식이야:

---
팀 회의록
날짜: [입력]
참석자: [입력]
---

안건:
1. [입력]
2. [입력]
...

결정사항:
- [입력]

다음 회의: [입력]
---

이 형식으로 내가 필요한 정보만 입력하면 자동으로 회의록이 완성되게 만들고 싶어."

GitHub Copilot이 방법을 제시하면:

"좋아. 이제 이걸 실행할 때 날짜는 자동으로 오늘 날짜가 들어가게 해줘."

"참석자 이름은 쉼표로 구분해서 입력하면, 자동으로 목록 형식으로 만들어줄 수 있을까?"

**핵심 포인트:**
- 템플릿의 고정/가변 부분을 명확히 구분
- 자동화할 수 있는 것(날짜 등)은 자동화하기
- 입력 방식을 단순하게 만들기

### 예제 3: 할 일 목록 우선순위 정리

**문제 상황:**
할 일을 메모장에 적어두는데, 개수가 많아지면서 뭐부터 해야 할지 헷갈립니다. 각 항목마다 중요도와 마감일을 적어뒀는데, 이걸 보기 좋게 정리하고 싶습니다.

**4대 원리 적용:**

분해:
- 할 일 목록 읽어오기
- 중요도와 마감일 정보 추출하기
- 정렬 기준 적용하기
- 정리된 목록 출력하기

패턴 인식:
- 중요도 표시 방식의 패턴 (★★★, [중요], 높음 등)
- 마감일 표시 형식 (2025-11-20, D-3 등)
- 완료된 항목 표시 방식

추상화:
- 모든 세부사항보다는 우선순위에 집중
- 당장 급한 것과 나중에 할 것 구분
- 핵심 정보만 표시

알고리즘적 사고:
1. 텍스트 파일 읽기
2. 각 항목의 중요도와 마감일 파악하기
3. 마감일 임박 순으로 정렬하기
4. 같은 마감일이면 중요도 순으로 정렬하기
5. 정리된 목록 만들기

**GitHub Copilot과의 협업:**

"할 일 목록이 텍스트 파일로 있어. 각 줄이 하나의 할 일인데, 이런 형식이야:

- 보고서 작성 [중요] (2025-11-15)
- 이메일 답장 [보통] (2025-11-13)
- 회의 준비 [중요] (2025-11-14)

이걸 마감일이 가까운 순서로 정렬하고, 같은 날이면 [중요]가 위로 오게 하고 싶어."

"좋아. 이제 오늘 날짜 기준으로 3일 이내 마감인 것들을 빨간색이나 강조 표시로 구분해줄 수 있을까?"

"완벽해. 마지막으로 이미 완료한 항목은 [완료] 표시가 있는데, 이건 목록 맨 아래로 보내줘."

**핵심 포인트:**
- 데이터 형식을 정확히 설명하기
- 정렬 기준을 명확히 하기
- 단계적으로 기능 추가하기

### 예제 4: 학습 노트 체계화하기

**문제 상황:**
온라인 강의를 들으면서 메모를 많이 했는데, 여러 파일에 흩어져 있습니다. 중요한 개념, 예제 코드, 참고 링크 등이 섞여 있어서 나중에 복습하기가 어렵습니다.

**4대 원리 적용:**

분해:
- 노트에서 개념 설명 부분 찾기
- 예제 부분 분리하기
- 참고 자료 링크 정리하기
- 주제별로 카테고리 만들기

패턴 인식:
- 개념 설명의 공통 형식 찾기
- 예제 코드의 표시 방법 패턴
- 참고 링크의 위치 패턴
- 주제 구분 방법 찾기

추상화:
- 핵심 개념만 추출
- 중요 예제만 선별
- 필수 참고 자료만 남기기
- 복습에 필요한 정보에 집중

알고리즘적 사고:
1. 모든 노트 파일 읽어오기
2. 내용 유형별로 분류하기
3. 주제별로 재구성하기
4. 복습용 문서로 정리하기

**GitHub Copilot과의 협업:**

"여러 개의 마크다운 파일에 학습 노트가 있어. 각 파일에는 개념 설명, 예제, 참고 링크가 섞여 있는데, 이걸 주제별로 정리하고 싶어. 개념은 '## 개념' 헤딩 아래에, 예제는 코드 블록으로, 링크는 '[참고]'로 시작하는 줄에 있어."

"좋아. 이제 각 주제별로 별도 파일을 만들되, 모든 개념을 먼저 정리하고, 그 다음 관련 예제들을 모아서 보여줘."

"완벽해. 마지막으로 각 파일의 시작 부분에 그 주제에 대한 간단한 요약을 추가해줘. 핵심 개념 3-5개 정도로."

**핵심 포인트:**
- 정보의 패턴을 명확히 파악하기
- 재구성할 형식을 구체적으로 설명하기
- 단계별로 정리하며 진행하기

## 협업 패턴과 베스트 프랙티스

여러분이 실습하면서 느꼈겠지만, GitHub Copilot과 효과적으로 협업하는 데는 몇 가지 패턴이 있습니다. 이런 패턴들을 익혀두면 더 빠르고 정확하게 원하는 결과를 얻을 수 있습니다.

### 명확한 시작, 단계적 진행

가장 중요한 원칙은 "명확한 시작, 단계적 진행"입니다. 처음에 전체 상황을 명확하게 설명하되, 실행은 작은 단계로 나누어 진행하는 것입니다.

**좋은 시작 예시:**
"엑셀 파일에 판매 데이터가 있어. A열은 날짜, B열은 상품명, C열은 수량, D열은 금액이야. 이 데이터로 월별 판매 실적을 분석하고 싶어."

이렇게 시작하면 GitHub Copilot은 전체 맥락을 이해하고, 첫 단계를 제안할 수 있습니다.

**피해야 할 시작 예시:**
"판매 실적 분석 좀 해줘."

이건 너무 막연합니다. 어떤 데이터인지, 어떤 형식인지, 무엇을 원하는지 GitHub Copilot이 알 수 없습니다.

### 중간 결과 확인하기

각 단계마다 결과를 확인하고, 맞는지 틀린지 GitHub Copilot에게 피드백을 주세요. 이것이 매우 중요합니다.

**효과적인 피드백:**
"좋아, 데이터를 잘 읽어왔어. 그런데 첫 줄은 제목이라서 제외해야 해."
"완벽해! 이제 다음 단계로 넘어가자."
"이건 내가 원한 게 아닌데. 나는 카테고리별이 아니라 날짜별로 묶고 싶었어."

이런 피드백을 받으면 GitHub Copilot은 다음 제안을 더 정확하게 할 수 있습니다.

### 예시를 함께 제공하기

추상적인 설명보다는 구체적인 예시를 함께 주면 GitHub Copilot이 훨씬 정확하게 이해합니다.

**예시 없이:**
"날짜 형식을 보기 좋게 바꿔줘."

**예시와 함께:**
"날짜가 지금 '20251113' 형식인데, '2025-11-13' 형식으로 바꿔줘."

후자가 훨씬 명확하고, 원하는 결과를 얻을 가능성이 높습니다.

### 제약 조건 명시하기

중요한 제약 조건이나 주의사항이 있다면 미리 말해주세요.

**제약 조건 예시:**
"원본 파일은 절대 삭제하거나 수정하지 말고, 새 파일로 만들어줘."
"금액이 0원인 항목은 제외해줘."
"파일 이름에 특수문자가 들어가면 안 돼."

이런 조건을 미리 알려주면, 나중에 문제가 생기는 것을 방지할 수 있습니다.

### 실패를 두려워하지 않기

GitHub Copilot의 첫 번째 제안이 항상 완벽한 것은 아닙니다. 그리고 그건 전혀 문제가 아닙니다. 시도해보고, 안 되면 다시 설명하고, 개선하는 과정이 바로 바이브 코딩입니다.

**효과적인 재요청:**
"이 방법은 파일이 많을 때 너무 느린데, 더 빠른 방법이 있을까?"
"이렇게 하면 중복된 항목이 생기네. 중복을 제거하는 방법을 추가해줘."
"거의 다 됐는데, 결과를 파일로 저장하는 부분이 빠졌어."

이렇게 구체적으로 문제를 지적하면, GitHub Copilot은 개선된 방법을 제시합니다.

### 코드를 이해하려 하지 말고 결과를 확인하기

일반인편의 핵심 원칙을 기억하세요: 생성된 코드를 읽고 이해할 필요는 없습니다. 중요한 것은 "내가 원한 결과가 나왔는가"입니다.

GitHub Copilot이 제시한 방법을 실행했을 때:
- 파일이 제대로 정리되었나요?
- 계산 결과가 맞나요?
- 예상한 형식으로 나왔나요?

이것만 확인하면 됩니다. 그 과정에서 어떤 코드가 사용되었는지는 알 필요가 없습니다. 바이브 코딩의 핵심은 "생각을 전달하고 결과를 확인하는 것"이지, 코드를 작성하고 이해하는 것이 아닙니다.

코드를 보면 어렵게 느껴질 수 있습니다. 낯선 단어들, 기호들, 복잡해 보이는 구조들이 여러분을 주눅들게 만들 수 있죠. 하지만 걱정하지 마세요. 요리할 때 요리사의 손놀림을 정확히 이해하지 못해도 맛있는 음식을 먹을 수 있는 것처럼, 바이브 코딩에서도 코드의 세세한 부분을 이해하지 못해도 문제를 해결할 수 있습니다.

만약 결과가 예상과 다르다면, 코드를 뜯어보려 하지 말고 GitHub Copilot에게 다시 설명하세요. "결과가 이렇게 나왔는데, 나는 이렇게 되길 원했어"라고 말하면 됩니다. GitHub Copilot이 알아서 수정해줄 것입니다.

### 재사용을 위한 문서화

한 번 잘 동작하는 방법을 찾았다면, 나중을 위해 기록해두세요. GitHub Copilot에게 이렇게 요청할 수 있습니다:

"이 과정을 다음에도 쉽게 따라 할 수 있게, 단계별로 정리해줘."

또는:

"이걸 다른 폴더나 파일에도 적용하려면 어떤 부분을 바꿔야 하는지 알려줘."

이렇게 하면 비슷한 문제를 만났을 때 처음부터 다시 시작하지 않아도 됩니다.

## 흔히 마주치는 어려움과 해결법

바이브 코딩으로 실생활 문제를 해결하다 보면 누구나 몇 가지 어려움을 겪게 됩니다. 이것은 전혀 이상한 일이 아닙니다. 오히려 자연스러운 학습 과정입니다. 여기서는 가장 흔하게 마주치는 어려움들과 그 해결 방법을 살펴보겠습니다.

### 어려움 1: "내가 원하는 걸 어떻게 설명해야 할지 모르겠어요"

이것은 아마도 가장 많이 겪는 어려움일 것입니다. 머릿속으로는 뭘 원하는지 알겠는데, 그걸 GitHub Copilot에게 어떻게 말해야 할지 막막한 경우죠.

**해결 방법:**

먼저, 완벽한 설명을 하려고 애쓰지 마세요. 일단 여러분이 생각하는 대로 말해보는 것이 중요합니다. "엑셀 파일을 정리하고 싶어"처럼 막연하게 시작해도 괜찮습니다. GitHub Copilot이 질문을 할 것이고, 그 질문에 답하다 보면 여러분도 원하는 것이 더 명확해집니다.

두 번째 방법은 예시를 사용하는 것입니다. "지금은 이런 모양인데, 이렇게 바꾸고 싶어"라고 구체적인 예를 들어주면 됩니다. 추상적으로 설명하는 것보다 훨씬 쉽고 정확합니다.

세 번째는 단계별로 나누는 것입니다. 전체를 한 번에 설명하려 하지 말고, "먼저 이것부터 해보자"라고 작은 단계로 시작하세요. 첫 단계를 성공하면 다음 단계는 훨씬 설명하기 쉬워집니다.

### 어려움 2: "GitHub Copilot이 엉뚱한 답을 줘요"

GitHub Copilot의 제안이 여러분이 원하는 것과 전혀 다를 때가 있습니다. 이럴 때 좌절하기 쉽지만, 사실 이것은 여러분의 잘못도, GitHub Copilot의 잘못도 아닙니다. 단지 의사소통이 아직 잘 안 된 것뿐입니다.

**해결 방법:**

화를 내거나 포기하는 대신, 다시 설명해보세요. 이번에는 조금 다르게, 더 구체적으로 말해봅니다. "아니야, 내가 원한 건 그게 아니라..."라고 시작해서, 무엇이 다른지 설명하세요.

또한 예상 결과를 명확히 해주는 것도 도움이 됩니다. "이렇게 해줘"보다는 "이렇게 하면 이런 결과가 나와야 해"라고 말하는 거죠. 결과를 구체적으로 설명하면 GitHub Copilot이 의도를 더 잘 이해합니다.

그리고 중요한 것은, 첫 번째 시도에서 완벽하기를 기대하지 않는 것입니다. 대화하듯이 주고받으면서 점점 개선해나가는 것이 바이브 코딩의 자연스러운 과정입니다.

### 어려움 3: "결과가 맞는지 틀린지 확신이 안 서요"

코드를 이해하지 못하니까, GitHub Copilot이 만든 결과가 정말 맞는 건지 불안할 수 있습니다. 특히 숫자 계산이나 데이터 정리 같은 경우, 겉으로는 그럴듯해 보여도 실제로는 틀릴 수 있죠.

**해결 방법:**

가장 좋은 방법은 작은 샘플로 테스트하는 것입니다. 전체 데이터에 바로 적용하지 말고, 몇 개 항목만 가지고 먼저 해보세요. 그럼 결과를 직접 눈으로 확인할 수 있습니다.

예를 들어 천 개의 파일을 정리하는 작업이라면, 일단 열 개 파일로만 테스트해보는 거죠. 그 열 개가 제대로 정리됐다면, 나머지도 같은 방식으로 잘 될 가능성이 높습니다.

또 다른 방법은 결과를 여러 방식으로 검증하는 것입니다. 예를 들어 금액 합계를 계산했다면, 계산기로 직접 몇 개 항목을 더해보세요. 계산 결과가 일치하나요? 그렇다면 나머지도 맞게 계산되었을 것입니다.

그리고 GitHub Copilot에게 검증을 요청할 수도 있습니다. "이 결과가 맞는지 확인하는 방법을 알려줘"라고 물어보세요. GitHub Copilot이 검증 방법을 제시해줄 것입니다.

### 어려움 4: "너무 복잡해서 어디서부터 시작해야 할지 모르겠어요"

문제가 너무 크고 복잡해 보일 때가 있습니다. 여러 단계가 얽혀 있고, 고려할 것이 많아 보이고, 어디서부터 손을 대야 할지 막막합니다.

**해결 방법:**

이럴 때야말로 컴퓨팅 사고의 '분해'가 빛을 발합니다. 큰 문제를 억지로 한 번에 해결하려 하지 말고, 가장 작은 부분부터 시작하세요.

"전체적으로 이런 걸 하고 싶은데, 지금은 일단 첫 단계만 도와줘"라고 GitHub Copilot에게 말하세요. 첫 단계가 성공하면 자신감이 생기고, 다음 단계도 할 수 있을 것 같은 기분이 듭니다.

또한 완벽을 추구하지 마세요. "일단 대충이라도 작동하는 버전"을 먼저 만들고, 그 다음에 조금씩 개선해나가는 것이 훨씬 효과적입니다. 처음부터 모든 경우를 다 처리하려 하지 말고, 가장 일반적인 경우만 먼저 해결하세요.

### 어려움 5: "시간이 너무 오래 걸려요"

바이브 코딩으로 하면 금방 끝날 줄 알았는데, 생각보다 시간이 많이 걸릴 때가 있습니다. 여러 번 시도하고, 수정하고, 다시 시도하다 보면 직접 하는 게 빠를 것 같다는 생각이 들 수도 있습니다.

**해결 방법:**

첫 번째는 인내심입니다. 처음 하는 것이라면 시간이 걸리는 것이 당연합니다. 하지만 한 번 방법을 익히면, 다음에는 훨씬 빠르게 할 수 있습니다. 그리고 비슷한 문제를 만났을 때는 이미 배운 패턴을 재사용할 수 있어서 정말 빠릅니다.

두 번째는 목표 조정입니다. 처음 생각했던 것보다 범위를 줄여보세요. "모든 걸 자동화"하려 하지 말고, "가장 귀찮은 부분만 자동화"하는 것으로 시작하세요. 80%의 효과를 20%의 노력으로 얻는 것이 현명합니다.

세 번째는 학습으로 생각하는 것입니다. 지금 하는 이 작업 자체보다, 바이브 코딩하는 방법을 배우는 것이 진짜 목표라고 생각해보세요. 그렇게 보면 시간이 아깝지 않습니다. 이번에 배운 것은 평생 사용할 수 있으니까요.

### 어려움 6: "GitHub Copilot이 제안한 방법이 작동하지 않아요"

가끔 GitHub Copilot이 제안한 방법을 그대로 따라 했는데 오류가 나거나 작동하지 않을 때가 있습니다. 이럴 때 당황하기 쉽지만, 이것도 해결할 수 있습니다.

**해결 방법:**

먼저 오류 메시지를 GitHub Copilot에게 그대로 보여주세요. "이렇게 했더니 이런 오류가 났어"라고 말하면, GitHub Copilot이 문제를 파악하고 수정 방법을 제시합니다.

또한 여러분의 환경을 설명하는 것도 중요합니다. "윈도우를 쓰고 있어", "엑셀 2016 버전이야" 같은 정보가 문제 해결에 도움이 될 수 있습니다.

그리고 다른 방법을 물어보는 것도 좋습니다. "이 방법 말고 다른 방법은 없을까?"라고 하면, GitHub Copilot이 대안을 제시해줄 수 있습니다. 문제를 해결하는 방법은 항상 여러 가지가 있습니다.

## 실습 결과 요약

이번 주차에서 여러분은 단순히 예제를 따라하는 것을 넘어, 실제 여러분의 문제를 바이브 코딩으로 해결하는 경험을 했습니다. 이 과정에서 몇 가지 중요한 것들을 배우셨을 것입니다.

### 핵심 학습 내용

**실생활 문제를 바이브 코딩 관점으로 보는 법:**
처음에는 막연해 보이던 문제들이, 컴퓨팅 사고의 4대 원리를 적용하면 명확하고 해결 가능한 문제로 변환된다는 것을 경험했습니다. "정리가 안 된다"는 막연한 고민이 "이 데이터를 이런 기준으로 분류하고, 이런 순서로 정렬하자"는 구체적인 계획이 됩니다.

**4대 원리의 통합적 적용:**
분해, 패턴 인식, 추상화, 알고리즘적 사고는 따로따로가 아니라 함께 작동합니다. 문제를 작은 단계로 나누고(분해), 반복되는 규칙을 찾고(패턴 인식), 핵심에 집중하고(추상화), 순서를 정하는(알고리즘적 사고) 과정이 자연스럽게 연결됩니다.

**GitHub Copilot과의 효과적인 협업:**
단계적으로 진행하고, 중간 결과를 확인하고, 구체적인 피드백을 주는 것이 얼마나 중요한지 배웠습니다. 완벽한 첫 요청을 하려고 애쓰는 것보다, 작은 단계로 나누어 대화하듯 진행하는 것이 더 효과적입니다.

**코드가 아닌 사고의 중요성:**
바이브 코딩의 진짜 핵심은 코드 작성 능력이 아니라 문제를 명확하게 생각하는 능력이라는 것을 체감했습니다. 여러분이 문제를 명확하게 정리할 수 있다면, GitHub Copilot은 그것을 실행 가능한 형태로 만들어줄 수 있습니다.

### 실전 활용 팁

**작게 시작하세요:**
첫 바이브 코딩 프로젝트로 너무 거창한 것을 선택하지 마세요. 10분 안에 해결할 수 있는 작은 불편함부터 시작하는 것이 좋습니다. 성공 경험이 쌓이면 자연스럽게 더 복잡한 문제로 확장할 수 있습니다.

**실패를 학습 기회로:**
GitHub Copilot의 제안이 한 번에 완벽하게 작동하지 않을 수도 있습니다. 그럴 때 포기하지 말고, "왜 안 됐을까?"를 생각해보세요. 내가 설명을 덜 명확하게 했을까? 중요한 정보를 빼먹었을까? 이런 생각 과정 자체가 컴퓨팅 사고를 키웁니다.

**패턴을 기록하세요:**
잘 작동한 요청 방식, 효과적이었던 설명 방법을 메모해두세요. 비슷한 문제를 만났을 때 이 패턴을 재사용할 수 있습니다. "지난번에 파일 정리할 때 이렇게 설명했더니 잘 됐는데, 이번에도 비슷하게 해볼까?"

**결과를 공유하세요:**
가족이나 동료에게 여러분이 바이브 코딩으로 해결한 문제를 보여주세요. "이걸 이렇게 자동화했어"라고 설명하는 과정에서 여러분의 이해가 더욱 깊어집니다. 그리고 그들도 "나도 이런 거 해결하고 싶은데"라고 말할 수 있습니다.

### 다음 단계를 위한 준비

이제 여러분은 바이브 코딩의 실전 활용을 경험했습니다. 다음 주차부터는 이 경험을 바탕으로 더 큰 프로젝트를 기획하고 실행하게 됩니다. 12주차와 13주차에서는 여러분만의 바이브 코딩 프로젝트를 처음부터 끝까지 완성하게 됩니다.

그 준비를 위해 지금부터 생각해볼 수 있습니다:

- 여러 사람에게 도움이 될 만한 솔루션을 만들 수 있을까요?
- 지금까지 배운 것들을 어떻게 조합하면 더 멋진 결과를 만들 수 있을까요?

바이브 코딩은 단순히 기술이 아니라 새로운 사고방식입니다. 여러분은 이제 "이건 어떻게 할까?"가 아니라 "이건 어떻게 설명할까?"를 먼저 생각하게 될 것입니다. 그리고 그 생각을 GitHub Copilot에게 전달하면, 놀라운 결과를 만들어낼 수 있습니다.

다음 주차에는 여러분만의 프로젝트 주제를 선정하고 구체적으로 기획하는 시간을 가지겠습니다. 이번 주차의 실습 경험을 바탕으로, 더 크고 의미 있는 도전을 준비해봅시다.
